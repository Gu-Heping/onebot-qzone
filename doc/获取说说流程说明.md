# 获取说说（getEmotionList）流程与「获取到 0 条」说明

## 一、整体调用链

```
API get_emotion_list / 轮询 pollMyPosts
  → client.getEmotionList(uin, pos, num)
    → 1) emotion_cgi_msglist_v6（proxy-json → proxy-jsonp → h5-json）
    → 2) 若全部返回 -10000（限流）或失败
         → getEmotionListViaFeeds3(uin, num, pos, maxPages)
```

- **emotion_cgi_msglist_v6**：官方 JSON 接口，易被限流（code=-10000「使用人数过多」）。
- **getEmotionListViaFeeds3**：用 feeds3 HTML 接口拉取并解析，是限流后的兜底路径。

---

## 二、getEmotionListViaFeeds3 内部策略顺序（拉「自己的说说」时）

| 顺序 | 策略 | 请求 | 说明 |
|------|------|------|------|
| 1 | scope=1 个人说说 | `fetchFeeds3Html(uin, true, 1, 50)` | 个人空间模式，自己时通常有数据 |
| 2 | （自己时无策略 2） | — | 好友时才有 scope=0+uinlist |
| 3 | scope=0 无 uinlist | `fetchFeeds3Html(uin, true, 0, 50)` | 好友动态流，再按 uin 过滤 |

任一策略返回的 `msglist.length > 0` 后，会用该策略的 `usedScope` 做**翻页**（见下）。

---

## 三、翻页逻辑（为何会出现「获取到 0 条」）

1. 首轮：用上面某策略拉第一页，得到 `msglist`。
2. 若 `msglist.length < pos + num`，则用 **externparam**（从当前页 HTML 里解析出的翻页参数）请求下一页：
   - `fetchFeeds3Html(..., externparam)` 再拉一页 HTML。
   - 对该页调用 `parseFeeds3Items(currentText, uin, undefined, 50)` 得到本页的 `page`（实现见 `src/qzone/feeds3/items.ts`）。
3. **「获取到 N 条说说」** 是**整次 getEmotionListViaFeeds3 结束时的汇总**：  
   `log('INFO', 'feeds3 fallback (${scopeLabel}) 获取到 ${msglist.length} 条说说')`。  
   这里的 `msglist` 已包含首轮 + 所有翻页合并、去重后的结果。

**单页解析为 0 的常见原因：**

- 该页 **HTML 里没有** `name="feed_data"` 的 `<i>` 标签，也没有解析器能识别的 JS 数据数组 → **feed_data 策略 0 条，JS fallback 也是 0 条**。
- 服务端返回的**不是正常列表页**：例如「没有更多」或错误页，响应体很小（日志里出现过约 115 字节），结构不符合解析预期 → 整页解析为 0。
- **翻页到头**：下一页本身就没有数据，或 externparam 指向的页面结构不同（如空列表模板），解析结果自然为 0。

因此会出现：  
- 首页或某一页：**「获取到 6 条说说」**（该次请求/该页解析到 6 条）；  
- 翻页的某一页：解析到 0 条（`page returned 0 items`），然后 `msglist` 不再增加，可能最终汇总仍是 6 条，或若首页就是 0 则整次 **「获取到 0 条说说」**。

---

## 四、解析器为何会得到 0 条（parseFeeds3Items）

解析有两层：

1. **策略 1：feed_data**  
   - 匹配 `name="feed_data"` 的 `<i>` 及 `id="feed_{opuin}_{appid}_{typeid}_{timestamp}_..."` 的块。  
   - 若指定了 `filterUin`（当前用户 uin），只保留 `opuin === filterUin` 的块。  
   - 若该页 HTML 没有这些结构（或全是别人/广告），则 **feed_data 策略得到 0 条**。

2. **策略 2：JS data 数组 fallback**  
   - 用正则从 JS 里找 `appid/key/abstime/uin/nickname` 等。  
   - 若该页没有这段 JS 或格式变了，则 **JS fallback 也是 0 条**。

于是该页 **parseFeeds3Items 返回 0**。  
若整次 getEmotionListViaFeeds3 所有策略 + 所有翻页都得到 0，最终就会打 **「feeds3 fallback (scope=0) 获取到 0 条说说」**。

---

## 五、和你日志的对应关系

- **「feeds3 raw response length=115」**：该次请求返回体只有 115 字节，多半是空页或占位响应，解析不到 feed_data/JS → 该页 0 条。
- **「parseFeeds3Items: feed_data strategy found 0, trying JS data array」→「JS fallback found 0 items」**：这一页两种策略都未命中。
- **「feeds3 pagination: page returned 0 items, 0 new after dedup」**：翻页这一页解析为 0 条，且没有新 tid，循环可能在这里结束。
- **「feeds3 fallback (scope=0) 获取到 6 条说说」**：同一轮里另一请求（或首页）解析到了 6 条，最终合并后总共 6 条。
- **「feeds3 fallback (scope=0) 获取到 0 条说说」**：这一整次 getEmotionListViaFeeds3 所有策略和翻页合并后，总数为 0。

---

## 六、结论与建议

- **「获取到 0 条」** = 这一**整次** feeds3 兜底（可能多策略、多页）合并后，解析到的说说条数为 0；不是「你空间里没有说说」。
- 常见原因：  
  1）某页/某次返回的 HTML 过小或结构异常（如 115 字节）；  
  2）翻页到头或 externparam 对应页无数据；  
  3）QZone 该接口返回结构变化，导致 feed_data 与 JS 两种策略都匹配不到。
- 若**同一时段**另有日志出现 **「获取到 6 条说说」** 等，说明其他请求/页是正常的，只是部分请求或翻页会得到 0 条。
- 想进一步确认可：在 `client.ts` 里对 `fetchFeeds3Html` 返回的 `text.length` 或 `text.slice(0,200)` 打日志，区分「空/极短响应」与「大段 HTML 但解析为 0」两种情况。
